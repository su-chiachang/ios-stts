# Three-tab layout — `stts | stt | tts` (2026-07-19)

Restructured the app from a single conversation window into three focused tabs,
sharing one `ConversationEngine`. This doc records what was built and, more
importantly, the four gotchas found while debugging the `[stt]` tab — each cost
real time and is easy to hit again.

## What was built

| Tab | Purpose | Key entry points |
| --- | --- | --- |
| **stts** | The existing STT → LLM → TTS conversation demo, unchanged | `ConversationView` |
| **stt** | Import an mp3/wav/mp4 → whole-file transcript with timestamps, switchable **Sentence** (default) / **Word** | `SttFileView`, `ConversationEngine.transcribeFileTimestamped`, `ParakeetStt.transcribeFileWords`, `TranscriptSegmenter` |
| **tts** | Type text → speak it in the model's **default** voice or a **zero-shot cloned** voice (import clip or record 3–5 s on-device) | `TtsView`, `ConversationEngine.speak(_:referenceWavPath:)`, `VoiceClipRecorder` |

Container: `RootTabView` (new); `STTSApp` roots there and moves the one-time
`loadModels` task up. Native/shim layer unchanged — `[stt]` timestamps use the
already-declared `parakeet_capi_transcribe_pcm_batch_json_lang`.

### Scoping notes
- **TTS preset voices and voice design are out of scope** — the vendored
  `qwen3-tts.cpp` port only implements default synthesis + cloning (no preset
  selector, no text-prompt voice design). They appear only as a disabled
  "coming soon" affordance.
- **Timestamps**: per-word timing (`words[{w,start,end,conf}]`) comes straight
  from the native JSON; sentence grouping is derived in `TranscriptSegmenter`
  (punctuation + inter-word gap scaled by `frame_sec`).

## Gotchas found while debugging (the useful part)

### 1. `.fileImporter` silently no-ops → use `NSOpenPanel`
This app is AppKit-hosted (`@NSApplicationDelegateAdaptor`) and sandboxed.
SwiftUI's `.fileImporter` **does not reliably fire its completion handler**
here — picking a file did nothing, no callback, no error ("完全沒變").

**Rule: always use `NSOpenPanel` for file pickers** (as every other picker in
the app already did — `ConversationView`, `SettingsView`). Panel-selected URLs
are authorized for the process by the `user-selected.read-only` entitlement, so
engine code reads them directly — no `startAccessingSecurityScopedResource` /
temp-copy dance needed. (Commit `2205394`.)

### 2. File transcription must auto-detect language, not reuse the chat locale
`transcribeFileTimestamped` initially forced `AppSettings.sttLocale` (the
conversation tab's setting) onto arbitrary files. Forcing e.g. `en` onto Chinese
audio makes parakeet return an **empty transcript**
(`[{"text":"","words":[]}]`). The `[stt]` tab now passes `lang: nil`
(auto-detect), decoupled from Settings. (Commit `1ee1613`.)

### 3. Empty transcripts must not fail silently
An empty result reverted the view to its "choose a file" prompt — looks like
nothing happened. Empty results now finish in an error state with an
actionable message. (Commits `1ee1613`, `d1a6f20`.)

### 4. STT model matters: nemotron (multilingual) vs EOU-120m (English-only)
The real cause of "中文檔轉不出來" was the **wrong STT model selected in
Settings**. Empirically verified:

| Model | Languages | `<EOU>` events | Role |
| --- | --- | --- | --- |
| `nemotron-3.5-asr-streaming-0.6b-q8_0` | **Chinese + English** (near-WER-0) | ❌ never emits | Main ASR / **use this for file transcription** |
| `realtime_eou_120m-v1-q8_0` | **English only** | ✅ emits | Live-mic end-of-utterance turn-taking only |

Test result (headless, same files):

```
EOU-120m × jfk.mp3 (English)  → 22 words, ends with a literal <EOU> tag
EOU-120m × Chinese clip       → empty, zero words
nemotron × both               → correct in both languages
```

So `realtime_eou_120m-v1` **does** do ASR — but only English; it returns empty
for Chinese. It was kept in the project because it is the *only* model that
emits `<EOU>` (nemotron never does — see `progress.md` M0). **For `[stt]` file
transcription, always select nemotron in Settings.**

Note: the EOU model appends a literal `<EOU>` marker to its text.
`ParakeetStt.strippingLanguageTag` only strips `<zh-CN>`/`<en-US>`-style locale
tags, not `<EOU>`/`<EOB>` — so those would leak if the EOU model were ever used
for file transcription. Not fixed (nemotron is the right model); noted here in
case EOU-model English transcription is wanted later.
